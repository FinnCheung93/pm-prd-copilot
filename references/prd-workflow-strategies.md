# 新写 / 修改 / 增量 / 评审策略

动笔前先判断本次 PRD 作业类型。不要把新写、修改、独立增量和评审都走成同一种流程。

| 类型 | 适用场景 | 主要策略 |
| :--- | :--- | :--- |
| 新写 PRD | 从零写一个新功能、新模块或新版本需求 | 先过首次生成澄清闸门，再用默认骨架成文；重点防止需求不明时直接写完整 PRD |
| 修改已有 PRD | 旧文档仍是当前权威，需要修正、补充或改造其中规则 | 不走完整首次澄清；先确认修改意图、扫描影响范围、补线上逻辑缺口 |
| 独立增量 PRD | 基于旧能力做新版本、新策略或新模块，直接改旧文会造成历史口径混乱 | 先过澄清闸门，并额外确认「线上现状 → 本期变更」；新建独立文档承载本期变更 |
| 仅评审已有 PRD | 用户要“看看有没有问题”“过一下评审”“查漏补缺” | 不默认重写；若目标是后续修改，先按单问原则补信息；若用户明确要一次性报告，再输出问题归并、风险分级和待裁决项 |

## 新写 PRD 策略

新写时优先确认：

- 问题与目标：为什么做，解决什么问题，成功信号是什么。
- 目标用户与场景：谁用，在哪些场景用，核心路径是什么。
- 范围与非范围：本期做什么、不做什么，哪些是后续版本。
- 主流程与核心规则：入口、步骤、状态、异常、边界。
- 依赖与约束：端、版本、地区、权限、配置、接口或全局规则。

新写 PRD 的前置流程：

1. 读取原始需求和必要上下文；读取前只允许说明“先读取资料并做澄清门检查”，读取后先输出 `SKILL.md` 的澄清门检查，不直接生成完整 PRD。
2. 第一轮补齐明显关键缺口：按用户侧单问原则提出一个主问题，可用快选式问题降低用户输入成本。
3. 四角色第二轮检查：产品、研发、测试、用户作为内部检查视角；若仍需澄清，每轮只问一个主问题，不外显多问题清单。
4. 理解摘要确认 / 成文：只有关键槽位已足够清楚时，才先输出一句自然语言理解摘要，再进入默认骨架。

新写 PRD 默认使用精简骨架，正文顺序：

1. 修订记录。
2. 本期一句话。
3. 范围与变更清单。
4. 关键规则与逻辑。
5. 异常 / 边界 / 待确认（如有）。
6. 实现依赖（仅必要时）。

新写 PRD 的质量重点：

- 不从方案倒推问题；目标与用户价值要站得住。
- 背景只服务理解本期变更，不承担立项说服。
- 范围边界必须清楚，避免 P0/核心需求过多。
- 默认不生成独立背景、概述、功能列表、信息架构、流程图或附录；复杂需求才按需升级。
- 默认不生成配置项、埋点、指标或数据上报表；只有用户明确要求或不写会影响研发/测试落地时才补。
- 关键槽位不足时继续追问，不直接生成完整 PRD；即使用户要求按现有信息推进，也只能输出待确认摘要或最小补充问题，不能把假设写成完整 PRD。

## 修改已有 PRD 策略

修改旧文前先确认用户真实意图：

- 更新 / 精修：局部补充、修错、补异常或补字段。
- 重写 / 重构：旧文结构混乱，需要重组章节或规则。
- 评审：只指出问题，不直接改正文。
- 新版本 / 独立增量：基于旧文写下一版或新增能力。

适合直接改旧文的情况：

- 旧 PRD 仍是当前权威文档。
- 本次变更属于旧 PRD 定义能力的一部分。
- 改动不会造成历史版本理解混乱。
- 只是修正错漏、补充边界、补充字段、补异常流程或调整同一规则。

修改旧文的固定动作：

1. 找本次要改的核心规则。
2. 搜索同义表述、旧规则、相关字段、相关状态、相关流程。
3. 标记受影响章节、功能列表、流程图、必要数据定义、修订记录和关联文档。
4. 决定每处是修改、删除、移动、引用，还是保留不动。
5. 更新修订记录，摘要写结果而不是过程。

修改已有 PRD 的质量重点：

- 不强制走完整首次生成澄清闸门；只补影响修改准确性的缺口。
- 本期一句话只写本次变更影响，不复述旧 PRD 结构、背景或历史修订。
- 同一规则只保留一个权威位置；避免两个章节同时定义同一规则。
- 被替换的旧规则要删除、更新或改为引用，不能让新旧口径并存。
- 功能列表、正文、流程图、必要数据定义、修订记录要一致。
- 新旧逻辑差异要清楚，尤其是「线上现状 → 变更后」。
- 未受影响章节不做无关重写。

## 独立增量 PRD 策略

适合新建独立增量 PRD 的情况：

- 本次是新版本、新模块、新策略，影响范围相对独立。
- 旧 PRD 已经代表线上历史口径，不应直接改写。
- 改旧文会造成“到底以旧版还是新版为准”的双源冲突。
- 需要把本期变更与线上现状清楚分开。

独立增量 PRD 应写清：

- 本期目标与范围。
- 依赖的旧 PRD / 全局规格 / 接口文档。
- 旧规则中保持不变的权威口径。
- 本期新增或变更的规则。
- 与线上现状的差异、迁移、兼容或灰度要求（如有）。

独立增量可以先读取旧 PRD / 全局规格 / AB 灰度文档，但读取目的只限于确认「线上现状 → 本期变更」。读取后必须先输出澄清门检查，不直接写 PRD。

澄清门前禁止宣布新文件名、章节结构、正文写法、README/清单同步或任何生成/落盘承诺。读取前只允许说明“先读取资料并做澄清门检查”，不能说“我会新增 xx.md”“正文会写 xx”“我会同步 xx 清单”。

独立增量读取后必须检查是否还缺：

- 本期目标与范围。
- 目标用户与影响对象。
- 线上现状。
- 本期变更规则。
- 生效范围 / 灰度 / 地区 / 端。
- 异常边界。

任一缺口会影响 PRD 准确性、研发落地或测试验证时，先追问，不假设、不直接写 PRD。

独立增量 PRD 的前置流程：

1. 读取原始需求、旧 PRD、全局规格或 AB/灰度文档中必要资料。
2. 按 `SKILL.md` 固定模板输出澄清门检查：场景判断、已知信息、关键缺口、下一步。
3. 若需要澄清，按用户侧单问原则提出一个主问题；若无需继续澄清，进入产品 / 研发 / 测试 / 用户四角色第二轮检查。
4. 第二轮无重要问题后，输出理解摘要，再按默认骨架只写本次变更对宣讲和开发落地的影响；旧 PRD 长背景、旧规则细节和历史修订用链接指代，不复述。

## 仅评审已有 PRD 策略

仅评审时，不默认改写正文。若用户希望评审后继续修改，默认先按用户侧单问原则补齐会影响修改判断的信息。只有用户明确要求“一次性评审报告 / 问题清单 / 只评审不改”时，才批量输出：

- 阻断项、重要项、建议项。
- 重复问题归并结果。
- 关键缺口：目标、范围、规则、数据、依赖、异常、边界中哪些会影响 PRD 落地。
- 待裁决项。
- 建议修改方向。
- 应移出宣讲主线的内容，例如过长背景、历史修订、原型链接、Q&A、脚本配置、后续演进、纯引用说明。
- 可以合并、降级或删除的章节，例如摘要/背景/概述重复、版本范围/功能列表重复、流程图机械生成、附录无必要。

若发现整体结构混乱、核心流程不清或目标不明确，且用户目标是后续修改，先转化为一个主问题继续澄清；若用户明确要一次性评审报告，则可列出结构性问题并建议是否进入重写 / 重构。
