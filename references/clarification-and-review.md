# 确认、追问、自检与问题分级

## 1. 先确认再动手

信息不足时优先通过对话继续推进，不要默认中断让用户整理材料。Agent 先自己判断资料是否足够、风险是否可控；关键信息足够时可以直接写，信息不足或风险较高时再问。

### 1.1 新写 / 独立增量 PRD 的写前判断

新写 PRD 和独立增量 PRD，在动手写正文前先判断这些信息是否足够。不要求逐项问完，只看是否足够支撑准确成文和研发落地。

- 目标用户与场景：谁在什么场景使用或受影响。
- 业务目标：本期要解决什么问题，成功结果是什么。
- 本期范围 / 非范围：本期做什么、不做什么，哪些明确后置。
- 核心规则或主流程：入口、步骤、状态、分支、前置/后置条件。
- 输入 / 数据 / 状态：用户或系统提供什么信息，涉及哪些关键状态或历史数据；不默认展开字段清单。
- 期望输出或成功结果：用户看到什么，系统返回、写入或通知什么。
- 依赖与约束：端、版本、地区、权限、配置、接口、全局规格或既有 PRD。
- 边界与风险：异常、灰度、回滚、敏感数据、外部通知、写回、人工确认。

不需要填满所有信息；核心信息足够清楚时可以直接成文。若需求涉及写回、敏感数据、外部通知、凭证、客户数据、人工审批或改造类线上逻辑，相关边界必须先问清。

### 1.2 模糊输入拆解

当输入来自老板想法、用户反馈、会议纪要、竞品观察或其它很散的原始需求时，先把它当成待拆解材料，不要直接当成 PRD 正文素材。

内部先看：

- 用户是谁。
- 用户现在怎么解决。
- 真实问题是什么。
- 哪些是必须做。
- 哪些只是想象中的需求。
- 哪些信息还不能下结论。
- 当前输入里是否存在目标冲突、范围冲突、用户诉求与业务目标冲突。

这不是用户可见问卷。需要问用户时，仍然一次只问一个问题。

拆解后按用户本轮要求和当前已经稳定的判断决定写到什么程度。用户要求“粗一些”或主要判断仍在变化时，只沉淀当前结论和未定问题，不把分析过程扩写成完整 PRD。

### 1.3 复杂状态需求识别

当需求涉及多状态业务实体、多角色操作、逆向分支、自动状态变化或非法流转风险时，把状态变化作为内部检查角度，不默认生成状态机章节。

适合重点检查状态变化的场景：

- 核心交易或流程类实体，例如订单、售后单、退款单、审批流、工单。
- 有生命周期的业务对象，例如优惠券、活动、商品上下架、内容审核、用户账号状态。
- 多角色操作、逆向分支较多的功能，例如支付、取消、申诉、退换货。
- 状态会随时间自动变化的对象，例如超时取消、自动确认、会员到期降级。

内部先看：

- 是否需要枚举合法状态。
- 是否存在会触发状态变化的用户操作、系统事件或超时事件。
- 是否存在必须拦截的非法流转。
- 是否存在正常、异常、逆向、自动流转路径。
- 是否需要为研发和测试写清状态边界、前置条件和异常处理。

反向边界：纯展示、单步操作、无状态工具，以及仅 2-3 个简单状态或开关、收藏、关注这类轻量状态，不引入状态机章节，也不生成状态流转图。

### 1.4 读取资料后的用户可见输出

对新写 PRD / 独立增量 PRD，读取资料后不要直接承诺生成文件或正文结构。先自己判断信息是否足够：

- 信息足够时，可以直接继续写，不必为了流程感等待用户确认。
- 信息不足时，用“先确认再动手”的结构说明已知信息、关键缺口和下一步。
- 已知信息只写从输入和资料中确认的事实，不写推测。
- 关键缺口只保留会影响 PRD 准确性、研发实现或测试验证的问题；不要把“为了完整更好”列为缺口。

需要向用户说明时，使用：

```markdown
**先确认再动手**

场景判断：[新写 PRD / 独立增量 PRD / 修改已有 PRD / 仅评审 PRD]

已知信息：
- [只列影响 PRD 落地的事实。]

关键缺口：
- [只列会影响准确性、研发落地或测试验证的问题；没有则写“暂无关键缺口”。]

下一步：[信息足够则继续写；还缺信息则一次只问一个问题。]
```

这是写前判断，不是 PRD 正文。

### 1.5 一次只问一个问题

每轮只问一个主问题；问题由 agent 根据当前上下文自行选择。能用 A/B/C/D 降低用户输入成本时，优先用快选。

```markdown
**确认 Q[序号]：[问题主题]**

[一个主问题。]

A. [短选项 1]
B. [短选项 2]
C. [短选项 3]
D. 其他

可直接回复 A/B/C/D，也可以补充一句说明。
```

使用规则：

- 关键缺口优先用 A/B/C/D 快选；开放问题可以自然提问。
- 只有能降低用户输入成本时才给选项，不为形式强行快选。
- 用户只回复字母时，按该选项吸收信息并继续判断。
- 用户回复 `D` 或 `D + 补充说明` 时，只围绕当前问题再问一个具体补充点。
- 快选只服务确认过程，最终 PRD 不保留 A/B/C/D 的过程痕迹。

### 1.6 产品 / 研发 / 测试 / 用户视角检查

关键信息初步清楚后，用四个视角做内部检查。不要把四个视角的问题都展示给用户；如果仍需要补信息，合并后一次只问一个问题。

| 视角 | 检查重点 |
| :--- | :--- |
| 产品 | 目标、范围、非范围、优先级、版本边界是否仍有歧义 |
| 研发 | 关键状态、接口依赖、兼容、并发/幂等、状态合法性或非法流转是否需要补充；不默认补配置清单 |
| 测试 | 正常路径、异常路径、逆向路径、自动流转、边界值、空值、状态流转是否可验证 |
| 用户 | 真实场景、操作路径、状态反馈、文案理解和预期结果是否清楚 |

### 1.7 按回答状态选择追问方式

| 输入状态 | 追问策略 |
| :--- | :--- |
| 模糊笼统 | 先问清对象、范围、场景、规则分别是什么 |
| 有数据支撑 | 验证来源、统计口径、适用范围和是否过期 |
| 有逻辑冲突 | 先指出冲突点，再请用户裁决以哪个说法为准 |
| 较完整 | 补边界、异常、非目标、依赖和历史逻辑 |

复杂需求可按需从三个方向追问：

- **是什么**：定义、范围、端、角色、版本、地区、状态、字段。
- **为什么**：依据、目标、用户价值、与线上现状的关系。
- **如果发生例外**：异常、边界、灰度、回滚、权限不足、数据为空、历史数据不兼容。

简单需求在核心足够清楚后即可成文，不为了完整感继续追问。

### 1.8 最小补充清单

确实无法继续写时，只列会影响 PRD 准确性的最小问题：

| 缺口 | 为什么需要 | 推荐补充方式 |
| :--- | :--- | :--- |
| 线上现有流程 | 改造类需求不能假设现状 | 提供当前流程、状态、依赖或既有 PRD 链接 |

## 2. 定稿前自检

### 2.1 文档卫生

- [ ] 使用默认骨架；没有机械生成独立背景、概述、版本范围、功能列表、信息架构、流程图或附录。
- [ ] 顶部有且只有一段「本期一句话」，不分点，不扩写成小节清单。
- [ ] 背景没有单独展开；未展开市场、竞品、价值论证或数据故事。
- [ ] 已删与上文重复的段落；无换说法重复同一规则。
- [ ] 纯新增需求未硬写「现有逻辑 vs 新逻辑」对照专章；改造类若涉及线上既有逻辑则已对齐。
- [ ] 无必要空章节已去掉；章节与需求匹配。
- [ ] 呈现方式自然；简单页面或单条规则没有被硬做成表格，长文本没有被塞进表格，表格主要用于稳定行列维度、横向比较、字段/参数对照、矩阵判断或状态转移对照。
- [ ] 修订记录已更新；关联文档及须同步的全局规格/配置（若有）已写清结果。
- [ ] 修订记录、原型链接、Q&A、脚本配置、后续演进没有进入宣讲主线；需要留档时只放入按需补充材料。
- [ ] 流程图只在文字/表格说不清状态机、跨端链路或分支流程时出现；没有机械添加 Mermaid。
- [ ] 可选章节确实降低歧义或承载复杂规则，而不是增加篇幅；没有机械生成配置项、埋点、指标或数据上报表。
- [ ] 只有存在歧义、风险、外部依据或跨团队协同时才补解释；内部常识不重复说明。
- [ ] 名词表与正文用语一致；平台/地区「生效/不生效」无歧义。
- [ ] 复杂 UI 已引用可访问原型；外链可打开；无敏感凭证。
- [ ] 产物完整度符合用户当前需要和已经稳定的产品判断；局部修订没有重写无关内容，用户明确保护的内容没有被擅自修改。
- [ ] 讨论素材与最终 PRD 已按工作策略分开；最终 PRD 只保留已经确认且影响落地的结论。
- [ ] 页面文案是用户实际看到的文字，没有混入设计意图、操作说明或面向内部的解释。

### 2.2 四视角轻量自检

只检查与本需求相关的项；不要为了凑角色而增加无关内容。

| 视角 | 检查重点 |
| :--- | :--- |
| 产品 | 目标、范围、非目标、优先级、现有逻辑与新逻辑是否清楚 |
| 用户 | 目标用户、真实场景、操作路径、状态反馈、文案理解是否清楚 |
| 研发 | 关键状态、接口依赖、兼容性、并发/幂等、状态合法性或非法流转是否需要说明；不默认要求配置项 |
| 测试 | 正常路径、异常路径、逆向路径、自动流转、边界值、空值、状态流转是否可验证 |

设计、运营、合规不作为常驻角色；仅在需求涉及时触发：

- 设计：复杂交互、空/加载/错误/成功/禁用状态、全局设计规范。
- 运营：灰度、回滚、推送、运营后台；埋点/指标仅在用户明确要求或会影响验收时触发。
- 合规：个人信息、权限、支付、UGC、未成年人、跨境数据、协议变更。

### 2.3 状态变化表达边界

状态机是复杂业务状态的表达工具，不是默认章节或固定产物。

- 轻量状态用文字、列表或必要短表说明即可。
- 状态枚举表、触发事件列表、流转规则说明优先于状态流转图。
- 只有文字或表格无法清楚表达状态机、跨端链路或分支流程时，才补流程图。
- 不默认输出 Mermaid。
- 不把状态机说明、状态流转图、状态枚举表、触发事件列表做成固定四件套。

### 2.4 问题归并与分级

自检或改稿发现问题时，先合并重复项，再按影响程度整理：

| 等级 | 含义 |
| :--- | :--- |
| 必须处理 | 不解决会导致核心流程无法落地、线上逻辑失真、资金/安全/合规风险、技术不可实现且无替代方案 |
| 重要问题 | 不解决会显著影响质量或交付，例如异常路径、边界条件、关键状态、状态流转、必要依赖缺失 |
| 建议优化 | 修复后更好但不阻塞，例如文案优化、非核心体验细节、后续版本可补充内容 |

多视角意见冲突时，不替用户拍板；列为「待确认」，说明冲突双方、影响范围和建议用户确认的点。

评审或改稿时额外检查：

- 是否存在逻辑不一致。
- 是否存在状态未覆盖或非法流转未说明。
- 是否存在描述无法开发。
- 是否存在路径无法测试。
- 是否存在结论缺少依据。
