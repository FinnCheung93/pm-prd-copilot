# 澄清、自检与问题分级

## 1. 澄清策略：追问优先

信息不足时优先通过对话继续推进，不要默认中断让用户整理材料。只有关键事实缺失会导致 PRD 失真、改造类线上逻辑无法确认，或用户明确要求先整理材料时，才输出最小补充清单。

### 1.1 首次生成澄清闸门

新写 PRD 和独立增量 PRD 必须先过澄清闸门。未补齐会影响 PRD 准确性、研发落地或测试验证的关键槽位前，不生成完整 PRD。

首次生成的关键槽位是检查镜头，不是问卷；不要求逐项问完：

- 目标用户与场景：谁在什么场景使用或受影响。
- 业务目标：本期要解决什么问题，成功结果是什么。
- 本期范围 / 非范围：本期做什么、不做什么，哪些明确后置。
- 核心规则或主流程：入口、步骤、状态、分支、前置/后置条件。
- 输入 / 数据 / 状态：用户或系统提供什么信息，涉及哪些关键状态或历史数据；不默认展开字段清单。
- 期望输出或成功结果：用户看到什么，系统返回/写入/通知什么。
- 依赖与约束：端、版本、地区、权限、配置、接口、全局规格或既有 PRD。
- 边界与风险：异常、灰度、回滚、敏感数据、外部通知、写回、人工确认。

不要求填满所有槽位；核心槽位足够支持准确理解时，先输出一句自然语言理解摘要，再进入默认骨架成文。若需求涉及写回、敏感数据、外部通知、凭证、客户数据、人工审批或改造类线上逻辑，相关边界必须先问清。

### 1.2 资料读取后的固定输出

对新写 PRD / 独立增量 PRD，读取原始需求、既有 PRD、全局规格、灰度/AB 文档或其它必要资料后，不得直接生成完整 PRD。必须先输出 `SKILL.md` 中定义的固定 **澄清门检查** 模板。

使用规则：

- 这是澄清过程，不是 PRD 正文，不使用默认 PRD 骨架。
- 已知信息只写从输入和资料中确认的事实，不写推测。
- 关键缺口只保留会影响 PRD 正确性、研发实现或测试验证的问题；不要把“为了完整更好”列为缺口。
- 若存在阻断缺口，紧接着按 `SKILL.md` 的用户侧单问原则提出一个主问题；不要一边列缺口一边生成 PRD。
- 若无阻断缺口，按 `SKILL.md` 中定义的 **四角色第二轮检查** 做内部检查；第二轮无需继续澄清后，才输出理解摘要并成文。

### 1.3 第一轮：关键缺口快选追问

第一轮只补会影响 PRD 准确理解和落地的关键缺口。每轮只问一个主问题；问题由 agent 根据当前上下文自行选择。

关键缺口提问格式：

```markdown
**澄清 Q[序号]：[问题主题]**

[一个主问题。]

A. [短选项 1]
B. [短选项 2]
C. [短选项 3]
D. 其他

可直接回复 A/B/C/D，也可以补充一句说明。
```

使用规则：

- 关键缺口优先用 A/B/C/D 快选；非关键或需要用户自由描述的问题可以自然提问。
- 只有能降低用户输入成本时才给推荐选项；不要为了形式每轮都推荐。
- 用户只回复字母时，按该选项吸收信息并继续检查槽位。
- 用户回复 `D` 或 `D + 补充说明` 时，只围绕该槽位再追问一个具体补充点。
- 快选只服务澄清过程，最终 PRD 不保留 A/B/C/D 的过程痕迹。

### 1.4 第二轮：多角色检查

第一轮关键缺口补齐后，用固定四角色做轻量评审：产品、研发、测试、用户。

| 视角 | 第二轮检查重点 |
| :--- | :--- |
| 产品 | 目标、范围、非范围、优先级、版本边界是否仍有歧义 |
| 研发 | 关键状态、接口依赖、兼容、并发/幂等是否需要补充；不默认补配置清单 |
| 测试 | 正常路径、异常路径、边界值、空值、状态流转是否可验证 |
| 用户 | 真实场景、操作路径、状态反馈、文案理解和预期结果是否清楚 |

四角色检查用于帮助 agent 判断是否还需要澄清，不是要求把四个角色的缺口都展示给用户。若仍需澄清，按 `SKILL.md` 的用户侧单问原则提出一个主问题；若无需继续澄清，才进入“理解摘要 + 默认骨架成文”。不要为了凑角色提问。

### 1.5 按回答质量选择追问方式

| 输入状态 | 追问策略 |
| :--- | :--- |
| 模糊笼统 | 先要求定义清楚：对象、范围、场景、规则分别是什么 |
| 有数据支撑 | 验证来源、口径、适用范围和是否过期 |
| 有逻辑冲突 | 先指出冲突点，再请用户裁决权威口径 |
| 较完整 | 补边界、异常、非目标、依赖和历史逻辑 |

### 1.6 分层追问

复杂需求按需使用三层追问：

- **What**：定义是什么？范围是什么？涉及哪些端、角色、版本、地区、状态、字段？
- **Why**：为什么这样做？依据是什么？与目标、用户价值、线上现状的关系是什么？
- **What if**：如果异常、边界、灰度、回滚、权限不足、数据为空、历史数据不兼容，会发生什么？

每轮只问一个主问题；简单需求在核心足够清楚后即可成文，不为了完整感继续追问。

### 1.7 结构性问题优先

若材料整体目标不明、范围混乱、核心流程断裂或多处互相矛盾，不要先列长清单。澄清对话中先转化为一个主问题，并给出推荐回答方向；只有用户明确要求问题清单或一次性评审报告时，才批量列出结构性问题。

### 1.8 最小补充清单

确实无法继续写时，只列会阻塞 PRD 准确性的最小问题，格式建议为：

| 缺口 | 为什么需要 | 推荐补充方式 |
| :--- | :--- | :--- |
| 线上现有流程 | 改造类需求不能假设现状 | 提供当前流程、状态、依赖或既有 PRD 链接 |

## 2. 定稿前自检

### 2.1 文档卫生

- [ ] 使用默认骨架；没有机械生成独立背景、概述、版本范围、功能列表、信息架构、流程图或附录。
- [ ] 顶部有且只有一段「本期一句话」，不分点，不扩写成小节清单。
- [ ] 背景没有单独展开；未展开市场、竞品、价值论证或数据故事。
- [ ] 已删与上文重复的段落；无「换说法再说一遍」。
- [ ] 纯新增需求未硬写「现有逻辑 vs 新逻辑」对照专章；改造类若涉及线上既有逻辑则已对齐。
- [ ] 无必要空章节已去掉；章节与需求匹配。
- [ ] 修订记录已更新；关联文档及须同步的全局规格/配置（若有）已结论化。
- [ ] 修订记录、原型链接、Q&A、脚本配置、后续演进没有进入宣讲主线；需要留档时只放入按需补充材料。
- [ ] 流程图只在文字/表格说不清状态机、跨端链路或分支流程时出现；没有机械添加 Mermaid。
- [ ] 可选章节确实降低歧义或承载复杂规则，而不是增加篇幅；没有机械生成配置项、埋点、指标或数据上报表。
- [ ] 只有存在歧义、风险、外部依据或跨团队协同时才补解释；内部常识不重复说明。
- [ ] 名词表与正文用语一致；平台/地区「生效/不生效」无歧义。
- [ ] 含 Mermaid 时符合 **prd-mermaid-conventions**；含复杂 UI 时已用 **prd-prototype-embed** 或等价原型链接。
- [ ] 无敏感凭证；外链可打开。

### 2.2 四视角轻量自检

只检查与本需求相关的项；不要为了凑角色而增加无关内容。

| 视角 | 检查重点 |
| :--- | :--- |
| 产品 | 目标、范围、非目标、优先级、现有逻辑与新逻辑是否清楚 |
| 用户 | 目标用户、真实场景、操作路径、状态反馈、文案理解是否清楚 |
| 研发 | 关键状态、接口依赖、兼容性、并发/幂等是否需要说明；不默认要求配置项 |
| 测试 | 正常路径、异常路径、边界值、空值、状态流转是否可验证 |

设计、运营、合规不作为常驻角色；仅在需求涉及时触发：

- 设计：复杂交互、空/加载/错误/成功/禁用状态、全局设计规范。
- 运营：灰度、回滚、推送、运营后台；埋点/指标仅在用户明确要求或会影响验收时触发。
- 合规：个人信息、权限、支付、UGC、未成年人、跨境数据、协议变更。

### 2.3 问题归并与分级

自检或改稿发现问题时，先合并重复项，再按风险排序：

| 等级 | 含义 |
| :--- | :--- |
| 阻断项 | 不解决会导致核心流程无法落地、线上逻辑失真、资金/安全/合规风险、技术不可实现且无替代方案 |
| 重要项 | 不解决会显著影响质量或交付，例如异常路径、边界条件、关键状态、状态流转、必要依赖缺失 |
| 建议项 | 修复后更好但不阻塞，例如文案优化、非核心体验细节、后续版本可补充内容 |

多视角意见冲突时，不要替用户拍板；列为「待裁决项」，说明冲突双方、影响范围和推荐决策点。
