# 默认骨架与基础规则

面向 **移动 App / Web** 等产品、以**研发可落地**为目标的 PRD 写法参考。默认使用精简骨架，不再用长章节菜单驱动生成。

## 1. 默认骨架

默认 PRD 只使用以下骨架；与需求无关的块整块省略，额外章节只在能降低歧义时升级。

| 块 | 默认写法 |
| :--- | :--- |
| **修订记录** | 固定保留，作为文档维护区；宣讲时不讲历史修订。 |
| **本期一句话** | 固定一段自然段，不分点；说明本期做什么、影响谁、关键变化是什么。 |
| **范围与变更清单** | 合并版本、范围、功能/需求列表；用表格写模块/对象、本期变化、生效范围、不做什么。 |
| **关键规则与逻辑** | 只写新增、变更、废弃的规则；按模块写业务逻辑、处理流程、前置/后置条件、正常与异常场景。 |
| **异常 / 边界 / 待确认（如有）** | 仅存在异常、风险、歧义或待确认时写；不要为了完整感硬加。 |
| **实现依赖（仅必要时）** | 只有不写会影响研发落地时才写；可承载关键字段、状态、接口、全局规格或 HLD/API 链接。 |

## 2. 合并规则

- 「宣讲摘要 / 背景 / 概述」合并为 **本期一句话**；背景默认不单独成章，不展开市场、竞品、价值论证或数据故事。
- 「版本与范围 / 功能列表 / 需求列表」合并为 **范围与变更清单**；小需求可以只用一个短表。
- 「信息架构 / 入口权限导航矩阵 / 数据定义 / 非功能性需求 / 配置与接口」不作为默认章节；只有直接影响业务规则或研发实现时，才收敛到 **关键规则与逻辑** 或 **实现依赖（仅必要时）**。
- 「附录 / Q&A / 后续演进 / 完整外链 / 调研依据」默认不出现；只有用户明确要求，或存在歧义、风险、外部引用必须留档时才作为按需补充材料。
- 「配置项 / 埋点与指标 / 数据上报」默认不生成独立章节或表格；只有用户明确要求、现有 PRD 已有对应维护区，或不写会导致研发/测试无法落地时才补充。

## 3. 现有逻辑处理

当新需求涉及修改、扩展或替代现有功能/流程/历史逻辑时，若输入不完整，主动询问：

1. 现有功能的完整业务流程和规则。
2. 历史版本的逻辑变更记录（如有）。
3. 现有数据结构和状态定义。
4. 现有功能与其他模块的依赖关系。

改造类需求必须写清「线上现状 → 本期变更后」。纯新增需求不写「现有 \| 新增」对照表；用依赖文档链出权威口径，正文直接写新增规则、入口与数据。

沿用现有逻辑时，只写“沿用 xxx 文档/章节”及本期差异；不要为了宣讲完整而复制旧规则。

## 4. 流程图与可视化

流程图默认不生成。只有以下情况才画 Mermaid 或 ASCII：

- 状态机用文字或表格说不清。
- 跨端链路存在多方职责和回跳。
- 分支流程会影响研发、测试或上线判断。

若只是普通入口、跳转、展示/隐藏规则，优先用表格或规则列表，不画流程图。

## 5. 与全局规格及安全

- 若组织维护跨需求规格（入口矩阵、弹窗优先级、实验与灰度清单等），且本需求触及：在 PRD 或修订记录中写明是否已同步或待办；Markdown 互链若项目有统一约定，使用 **workspace-doc-links**（若启用）。
- 勿在 PRD 正文中粘贴访问令牌、Secret、Cookie、Webhook 密钥、协作工具同步的机器人 token 等；从外部工具粘贴内容前须脱敏。
- 技术调研、设计稿等外链保持可访问；失效时在修订记录中注明替代链接。
